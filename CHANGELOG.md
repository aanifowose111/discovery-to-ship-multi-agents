# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

This project does not yet follow strict semantic versioning. Pre-1.0, breaking changes happen as the methodology evolves. Once the system stabilizes through use, this will move to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

**Conventions for entries under `[Unreleased]`:**

- New batches added on a dev day where **no version has yet been cut** are grouped under a `### YYYY-MM-DD` subheader inside the Added / Changed / Fixed buckets so chronology stays visible before the version is cut.
- If **today already has a cut version section** (e.g., `## [0.4.0] - 2026-05-31`), do NOT duplicate the date under `[Unreleased]`. Instead, the new same-day batch should be cut as a **patch bump** (`v0.4.0 → v0.4.1`). Patch bumps cleanly handle same-day add-ons and concurrent-contributor scenarios (each push gets its own version), and avoid the confusing "two sections with the same date" layout.
- Cross-day work simply waits in `[Unreleased]` under its own date subheader until cut.

## [0.12.0] - 2026-06-09

Substantial expansion across the workflow: one-at-a-time questioning in the design picks, a Fijara resurface convention with seven trigger signals, two new macOS quality-of-life commands (caffeinate + break reminder), per-product GitHub push, granular CHECKLIST.md alongside BUILD_STATUS.md, and a deep-debug specialist for cross-cutting bugs.

### Added

- **`/push-project <slug>` slash command** — push individual products (`web-apps/<slug>/`, `mobile-apps/<slug>/`, `desktop-apps/<slug>/`) to their own **independent GitHub repositories**, separate from the parent workspace repo. The parent's `.gitignore` already ignores those folders → each nested folder can have its own `.git/`, remote, history, branches, GitHub Actions. Supports `--init` (creates stack-specific `.gitignore`, `README.md`, optional `LICENSE`, sets remote, initial commit + push), routine push, `--status`, `--remote <url>` updates. **Mandatory safety scans** before every commit: refuses staged `.env` files (only `.env.example` allowed), scans for AWS keys / GitHub tokens / Stripe keys / private keys / Slack tokens, refuses plain `--force` to `main` (requires verbose acknowledgment flag). Per-commit Co-Authored-By trailer ask per CLAUDE.md policy. Stack-specific `.gitignore` templates included for Flask, RN+Expo, PySide6.
- **`guides/product/project-git-methodology.md`** — full methodology for nested-git pattern: why it works (parent's `.gitignore` shields nested `.git/`), setup checklist, stack-specific `.gitignore` templates, per-commit safety scans, common gotchas (forgot to `cd`, accidentally ran `git init` in parent, `.env` got staged before noticing, pushed a secret by accident → recovery via `git filter-repo`).
- **`/generate-checklist <slug>` slash command** — generate (or regenerate) the fine-grained `CHECKLIST.md` for a product. Companion to `BUILD_STATUS.md`: where BUILD_STATUS tracks coarse subsystems, CHECKLIST tracks individual deliverables (3-8 per must-have, with file-path hints in italics for mtime-based auto-cross-out). Decomposes must-haves from MVP.md (and V1.md if green-lit-to-build), groups by user-visible feature rather than technical layer, initializes a Scope changes log for auditability.
- **`/read-checklist <slug>` slash command** — refresh `CHECKLIST.md`. mtime-cached: only inspects files modified since the last refresh (cheap by design). Auto-cross-out via file-hint matching; proposes additions for newly discovered work (capped at 5 per pass, user confirms each via `AskUserQuestion`); updates frontmatter `last-scanned-at` + `last-scanned-mtime`. Detects completion when all MVP-scope items are `[x]`.
- **Auto-refresh hook in `senior-software-engineer`** — after every `BUILD_STATUS.md` subsystem flips to `[x]`, the orchestrator runs an inline equivalent of `/read-checklist` (cross-outs only, no scope-discovery prompts). Silent — appends to Scope changes log with "Triggered by orchestrator after `<subsystem>` flip" annotation. Keeps CHECKLIST current without user intervention.
- **`guides/product/checklist-methodology.md`** — methodology for `CHECKLIST.md`: §2 format spec, §3 decomposition rules (3-8 deliverables per must-have, group by user-visible feature, file-path hints), §4 mtime-cached auto-cross-out, §5 scope discovery, §6 user-driven add/remove, §7 auto-refresh hook in the orchestrator, §8 when NOT to use, §9 pipeline integration.
- **`/caffeinate` and `/stop-caffeinate` slash commands** — macOS-only display + system sleep prevention. `/caffeinate` spawns `caffeinate -d -i` in the background via `nohup`, captures PID at `~/.claude/caffeinate.pid`, reports back. `/stop-caffeinate` reads PID, sends SIGTERM, removes pidfile. Idempotent (stop when nothing's caffeinated is a benign no-op). Graceful errors on Linux/Windows with platform-specific hints (`systemd-inhibit`, PowerShell power requests).
- **Break-reminder hook at `.claude/hooks/break-reminder.sh`** — `UserPromptSubmit` hook that injects a reminder via `hookSpecificOutput.additionalContext` when the session has been active >2 hours (throttled to 1 hour between subsequent reminders). State files: `~/.claude/.session-start`, `~/.claude/.last-break-reminder`. Claude surfaces via `AskUserQuestion` at the next natural conversation boundary (does NOT interrupt mid-task): "Worked ~Xh Ym. Break?" — options (a) Caffeinate (continue, reset timer) / (b) Pause (wrap up, suggest `/recollect`) / (c) Stop after X+Y / (d) Other. Reset manually any time: `rm -f ~/.claude/.session-start`. Disable: `chmod -x .claude/hooks/break-reminder.sh`.
- **`/deep-debug <slug> [focus-area]` slash command** — invoke the new `senior-debugging-engineer` for deep root-cause analysis on a cross-cutting failure (flaky test, backend+frontend race condition, deploy failure with non-obvious cause, intermittent production bug). The persona runs a hypothesis-driven 5-phase process (Reproduce → Hypothesize 2-4 candidates → Test each → Identify root cause → Recommend fix) and returns a structured report. **Does NOT write the fix** — the relevant `senior-X-engineer` does that, against the report. Optional focus-area argument narrows the investigation.
- **`senior-debugging-engineer` subagent at `.claude/agents/senior-debugging-engineer.md`** — 9th specialist persona. Lens: "Find the root cause, not the symptom." Refuses common rationalizations ("probably a race condition" without timing evidence; "restart the dev server"; "skip the test, move on"; stopping at first plausible hypothesis without testing alternatives). Locked output format: Reproduction (with N/10 reliability), Symptom verbatim, 2-4 hypotheses tested (each CONFIRMED / RULED OUT with cited evidence), Root cause (file:line, what's wrong, causal chain to symptom, intermittency conditions), Fix recommendation (minimal fix description + defensive additions + owner persona + architectural concern if any), Confidence HIGH/MEDIUM/LOW.

### Changed

- **`/draft-design-spec <slug>` and `/draft-design-brief <slug>`: one-question-at-a-time picks.** Previously both commands asked all 9 picks in a single structured-message block. Now each pick is its own sub-step using `AskUserQuestion` for multi-choice (visual direction → palette → typography → portfolio stance → dark mode → font loading) and free-text prompts with explicit wait for the rest. **Pair-aware pre-filtering** — when the user picks a visual direction in 1.1, the palette options in 1.2 are reordered to surface the research's pairing match first. Each command ends with a summary back to the user + "Draft now? / Revise a pick?" confirmation before invoking the researcher. Fijara-checkpoint annotation added (per the new CLAUDE.md rule below).
- **Fijara resurface convention in CLAUDE.md § Working style.** Expanded the existing "Fijara referral when the user signals struggle" bullet into a structured rule with **7 trigger signals in descending strength** ("I'm stuck" / "I don't know how to" 3+ times; same `/rework` cycle 3+ times; same subsystem `[>]` 3+ days; 3+ reviewer REJECT overrides in a row; frustration language; >1 week inactivity; same question 3+ times) + **once-per-session cap** via `[fijara-surfaced-this-session]` marker + **new resurface message format** (gray-italic hint: "Send `MVP.md` + `DESIGN_SPEC.md` to **support@fijara.com**, or create a project draft at **fijara.com** for the team to reach out"). Doesn't push — trust the user's initial pick; exception is genuine evidence of mismatch.
- **`senior-software-engineer.md`** updated: when flipping any `BUILD_STATUS.md` subsystem to `[x]`, **also refresh `CHECKLIST.md` inline** if it exists (mtime-scan + cross-out + Scope changes log row). Silent — no user prompt; user-driven scope additions stay in the explicit `/read-checklist` flow.
- **`.claude/settings.json`**: added `hooks.UserPromptSubmit` wiring to `.claude/hooks/break-reminder.sh` using `$CLAUDE_PROJECT_DIR/.claude/hooks/break-reminder.sh` path.
- **CLAUDE.md compressions** to keep under the 40k auto-load threshold: trimmed redundant phrasing in the Reviewer-assistants bullet, Stack-flexibility bullet, and Search-patterns subsection (no information loss; same rules tighter).
- **README.md / HELP.md / CLAUDE.md command listings** updated with all 7 new commands.
- **`scripts/run_tests.py`** — 9 new required-file checks. Test count: 137 → 156.

### Notes

- **Mid-build the break-reminder hook needs a session restart to load** (settings.json changes don't hot-reload). After this commit lands, close + reopen Claude Code to pick up the hook.
- **The auto-refresh in `senior-software-engineer.md`** only fires for products that have `CHECKLIST.md` — generated via `/generate-checklist <slug>`. For ops-audit-agent (the in-flight build), run `/generate-checklist ops-audit-agent` to get the initial checklist after the v0.12.0 changes land.
- **`/push-project` does NOT create the GitHub repo for you.** Create the empty repo first (via `gh repo create <name> --private` from inside the product folder, or via the GitHub web UI), then `/push-project <slug> --init` wires the local repo to it.
- **Why senior-debugger is a 9th specialist, not just a skill:** `debugging-and-error-recovery` (existing skill) handles the routine path. The 9th specialist applies only to cross-cutting bugs where the existing per-domain specialists each see only their slice — and where root-cause discipline (hypothesis-driven, evidence-cited) matters more than the speed of a fix. Most bugs never trigger it.

Soft override requires explicit acknowledgment ("I accept the AI-generic aesthetic risk"), recorded in frontmatter as `design-artifact-skipped: true` for auditability.

Core-file confirmation rule continues to bite for `CLAUDE.md`, `guides/`, etc.

The disambiguator for "please continue" when multiple products are mid-build: passing the slug picks which one to resume. Reads `BUILD_STATUS.md` to find `[>]` (in-progress) and `[ ]` (pending) subsystems, invokes `senior-software-engineer` in resumption mode (the orchestrator summarizes what's done, what's mid-flight, what's next, and proposes the next specialist to engage — no orientation re-ask). Handles `build-status: rework-in-progress` automatically. Edge-case stops: no `BUILD_STATUS.md` (suggests `/start-build`); all subsystems `[x]` (suggests `/ship-app`); MVP `shipped` but no V1.md (suggests `/scope-v1`).
- **`/recollect <slug>` slash command** — read-only "where are we" synthesis for a specific product. Walks every related artifact (idea card, validation report, scoping reports, MVP/V1 briefs, design research/spec/brief, designer handoff, BUILD_STATUS, team.json, INFRA_COST, source tree, audit-log entries) and emits a one-screen report: pipeline-state table, build progress, source-tree at-a-glance, team, recent activity, prose synthesis, and 2-4 plausible next actions specific to the actual state. Distinct from `/status` (workspace-wide) and `/continue-build` (actually invokes orchestrator).

## [0.10.2] - 2026-06-07

Same-day patch on top of v0.10.1.

### Added

- **`.github/workflows/release.yml`** — auto-create a GitHub Release whenever a `vX.Y.Z` tag is pushed to origin. The workflow extracts the matching `## [X.Y.Z] - YYYY-MM-DD` section from `CHANGELOG.md` and uses it verbatim as the release body. Idempotent: if a release for that tag already exists, the workflow updates its notes instead of erroring (handy when re-pushing a tag after fixing CHANGELOG content). Manual fallback recipe is documented inline in the workflow file. Triggers on tags matching `vX.Y.Z` or `vX.Y.Z.N` (the latter covers patch-of-patch edge cases the same-day-patch rule could produce).
- **Backfilled GitHub Releases for v0.7.0 → v0.10.1.** Each one carries the corresponding CHANGELOG section as the release body. Combined with the v0.10.2 workflow, the repo's Releases sidebar now shows the active release cadence; the latest-release badge displays on the homepage.

### Notes

- **Why bother with Releases for a non-binary repo?** This isn't a typical software product — there's no pip package or binary download. But Releases add three discoverability surfaces beyond plain tags: the "Latest release" badge on the repo homepage, a per-version share URL (more concise than CHANGELOG anchors), and an Atom/RSS feed for followers. All three help with the share-the-repo CTA added in v0.8.0's README work.
- **Workflow self-test.** The workflow won't fire when this commit lands on `main` — it triggers only on tag pushes (`refs/tags/v*.*.*`). The first proof of correctness will be the next tag (`v0.10.2` after this commit + `git tag v0.10.2 && git push --tags`); the Release for v0.10.2 will be auto-created and will contain this entry as the body.

- **Build-phase UX improvements — narrated orchestration + BUILD_STATUS.md + `/preview-product`:**
  - **Narrated handoffs.** `senior-software-engineer` persona updated with explicit "speak the team's handoffs out loud" pattern: before invoking a specialist, announce *who* is doing *what* next, *from* which input artifacts, *toward* which expected output, with a rough time estimate. After completion, announce what landed + artifact + decisions. Same pattern at every handoff between specialists. The goal: the user reads along and feels the senior engineering team actively working for them, not a black box.
  - **`BUILD_STATUS.md` per product** — a dynamic, product-specific checklist of build subsystems. **Generated dynamically** (not from a static template) by walking the brief's must-haves (auth/uploads/jobs/etc. only included if scoped), the chosen stack (Flask vs. Next.js vs. ...), and the standard build order. Owned by `senior-software-engineer`; specialists report completion, senior-software-engineer writes/updates. New methodology guide at `guides/product/build-status-methodology.md` covers generation, format, update protocol, and how `/status` surfaces it.
  - **`/preview-product <slug> [page-name]` slash command** — preview a product's UI in the browser. Auto-detects which mode is possible: **real preview** (the running app at `localhost:5000/<page>`, requires dev server up + dependencies wired) or **dummy preview** (Jinja template + fixture data via the `web-preview` skill, always possible if the template exists). Always tells the user which mode they got and why. If dummy mode, briefly mentions what would unblock real preview. Web only — mobile previews go through Expo Go / EAS preview builds. The `web-preview` skill itself updated with a "Dummy mode vs. real mode" section explaining its role within `/preview-product`.
- **Slug-uniqueness rule and tooling.** Slugs must be unique across `ideas/`, `ideas/killed/`, `web-apps/`, and `mobile-apps/`. New script `scripts/check_slug.py` enforces this (exit 0 = available; exit 1 = taken with details). Used by `new_idea_card.py` (imports `is_available()`) and `lint_pipeline.py` (new `slug.collision` error rule). CLAUDE.md has a new "Slug uniqueness" section codifying the rule.
- **8 senior-engineer personas added** at `.claude/agents/senior-*.md` to orchestrate the build phase:
  - `senior-software-engineer` — generalist orchestrator; asks orientation questions, routes to specialists, sequences subsystems.
  - `senior-system-design-engineer` — system shape, data flow, cross-cutting concerns, decisions deferred. Produces SYSTEM_DESIGN.md.
  - `senior-database-engineer` — schema, indexes, migrations. Produces SCHEMA.md.
  - `senior-backend-engineer` — ORM models, API contracts, endpoint implementation, business logic. Produces API_CONTRACT.md.
  - `senior-frontend-engineer` — UI implementation (Jinja+JS or RN), design-token integration, accessibility.
  - `senior-qa-engineer` — test coverage audits, integration tests at the seam, release-readiness.
  - `senior-devops-engineer` — deploy, CI/CD, observability, incident response.
  - `senior-security-engineer` — threat modeling, secure coding, auth design, infra hardening.
  Each persona's body lists which agent-skills it commonly invokes (TDD, code-review-and-quality, security-and-hardening, performance-optimization, etc.), so the senior personas compose with the agent-skills.
- **`/start-build <slug>` slash command added.** Entry point for the build phase. Invokes `senior-software-engineer`, which asks the three orientation questions: (1) for hybrid briefs, web/API first (recommended) vs. mobile first; (2) MVP scope (recommended) vs. fully-featured; (3) which subsystem to start with (database design recommended), with the full ordered list shown for override. After answers, proposes the right specialist persona for the first subsystem and the specific first task. The build then proceeds subsystem-by-subsystem, with `senior-software-engineer` routing at each handoff.
- **CLAUDE.md "Build orchestration" subsection** added under Pipeline orchestration & checkpoints. Names the personas, the standard build order (database → project tree → models → API contract → API implementation → auth → background jobs → frontend → tests → deploy → iterate), and clarifies that senior personas are roles and agent-skills are workflows the roles execute.
